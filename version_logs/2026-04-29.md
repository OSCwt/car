# 2026-04-29 旧车迁移标注与交接说明

## 1. 原本存在的问题

这个工程是从旧车工程迁移到新车后的工作区，不是一个纯新工程。旧车代码、新车当前使用代码、底层库代码和工程配置依赖混在一起，单纯按“看起来没用”删除文件风险很高。

当前主要问题包括：

- 旧车代码和新车代码混在同一批 `code/` 文件中，阅读时不容易判断哪条路径正在被新车使用。
- 有些旧车实现已经不用，甚至已经被注释掉，但对应 `.c/.h` 文件仍然参与编译或被间接包含。
- 有些文件不能直接删除，是因为还存在头文件聚合 include、全局变量、ISR 符号、函数声明、宏定义或工程整目录编译依赖。
- `main.h`、`common.h`、`Icar.h` 这类文件目前是强依赖入口，贸然精简会引发连锁编译错误。

## 2. 本次做了什么

本次只做低风险整理和说明，不改变控制行为：

- 给旧车遗留代码增加了 `LEGACY_OLD_CAR` 标记，说明这些代码来自旧车逻辑，新车当前不作为主路径使用。
- 给新车当前使用代码增加了 `ACTIVE_NEW_CAR` 标记，帮助区分当前主要执行路径。
- 给暂时不能删除的依赖增加了 `KEEP_FOR_BUILD` 说明，标明其仍被工程配置、头文件、全局变量或其他模块依赖。
- 新增根目录 `MIGRATION_NOTES.md`，集中说明旧车迁移背景、新车主路径、旧车遗留模块、隐式依赖、后续精简顺序和新手阅读顺序。

涉及标注的重点文件包括：

- `code/Icar.c`
- `code/Icar.h`
- `code/common.c`
- `code/common.h`
- `code/Gpio.c`
- `code/Gpio.h`
- `code/SasuTimer.c`
- `code/PID_debug.c`
- `code/main.h`

## 3. 本次没有做什么

为了保证工程仍然可编译、可下载、可调试，本次明确没有做以下事情：

- 没有删除任何文件。
- 没有移动任何文件。
- 没有修改 `motor` / `encoder` / `servo` / `pid` / `uart` / `isr` / `main` 的核心控制逻辑。
- 没有修改 `.cproject`、`.project`、`Debug/*/subdir.mk` 或任何工程配置文件。
- 没有重构核心控制链路，也没有把旧车逻辑强行拆走。

## 4. 当前结果

当前工程结构比整理前更容易阅读：

- 新车当前主路径和旧车遗留路径有了注释标记。
- 暂时不用但不能删的代码有了风险说明。
- 后续接手的人可以先看 `MIGRATION_NOTES.md`，再沿着 `main.h`、`cpu0_main.c`、`top.c`、`SasuTimer.c` 和各控制模块理解主流程。
- 后续可以逐步做条件编译隔离或真正精简，但不建议在未验证依赖前直接删除旧文件。

## 5. 后续建议

建议后续按保守顺序推进：

1. 先保持当前可编译版本，把迁移标注作为阅读和排查依据。
2. 再逐步解耦 `main.h`、`common.h`、`Icar.h` 这些强依赖文件，减少聚合 include 和全局变量牵连。
3. 对新车暂时不用但会影响编译的旧车模块，优先考虑空实现或条件编译隔离。
4. 每隔离一小块都先完整编译、下载、调试验证。
5. 最后确认没有 include、宏、全局变量、ISR 和工程配置依赖后，再考虑删除旧车不用代码。

当前阶段的原则仍然是：先稳住能编译、能下载、能调试的版本，再做小步精简。
